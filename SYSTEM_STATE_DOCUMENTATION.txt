DIRECTION LOGIC (CORRECTED):

    Motor 1: IN1=HIGH, IN2=LOW   → Left wheel forward

  LEFT TURN (Motor1=-128, Motor2=+128):
    Motor 1: Slower/Backward
    Motor 2: Faster/Forward
  
  RIGHT TURN (Motor1=+128, Motor2=-128):
    Motor 1: Faster/Forward
    Motor 2: Slower/Backward

PWM CONFIGURATION:
  Frequency: 5000 Hz
  Resolution: 8-bit (0-255)
  Motor1 Channel: 0 (PWM1 pin 25)
  Motor2 Channel: 1 (PWM2 pin 32)

KINEMATICS:
  Forward Speed = (RPM1 + RPM2) / 2
  Angular Velocity = (RPM2 - RPM1) * wheel_radius / wheel_separation
  Odometry Update Rate: ~1 Hz (based on encoder samples)

================================================================================
ROBOT FRAME & KINEMATIC MODEL
================================================================================

COORDINATE FRAMES:
  /map       - Global map frame (SLAM creates this)
  /odom      - Odometry frame (relative origin)
  /base_link - Robot center (between wheels)
  /tof_sensor- Time-of-flight distance sensor

KINEMATIC MODEL: Differential Drive
  - 2 independently controlled wheels
  - Point robot assumption (width negligible)
  - Non-holonomic constraints

ODOMETRY CALCULATION:
  Δx = Δdist × cos(θ)
  Δy = Δdist × sin(θ)
  Δθ = Δdist_right - Δdist_left / wheel_separation
  
  Where:
    Δdist = (left_dist + right_dist) / 2
    left_dist = encoder1_pulses × distance_per_pulse
    right_dist = encoder2_pulses × distance_per_pulse

================================================================================
TROUBLESHOOTING GUIDE
================================================================================

PROBLEM: Robot not responding to /cmd_vel
SOLUTION:
  1. Check serial connection: ls /dev/ttyACM1
  2. Verify ESP32 firmware running: pio device monitor --port /dev/ttyACM1
  3. Check robot_controller logs: grep robot_controller /home/ros/.ros/log/latest/*
  4. Restart system: Kill all nodes, rebuild, relaunch

PROBLEM: Motors moving in wrong direction
SOLUTION:
  1. Current fix: Direction logic reversed in motors_encoders.cpp (completed)
  2. If still wrong: Swap motor connectors or IN pins in code

PROBLEM: Encoder pulses not counted
SOLUTION:
  1. Check GPIO pins: GPIO 4 and 5 for pulled-up signals
  2. Verify ISR: Check interrupt handler in motors_encoders.cpp
  3. Test encoder directly: Run encoder_test.cpp

PROBLEM: SLAM not creating map
SOLUTION:
  1. Verify /scan topic publishing: ros2 topic echo /scan
  2. Check scan data quality (should have non-zero ranges)
  3. Verification /tf tree has odom→base_link transform
  4. Move robot slowly in figure-8 pattern for loop closure

PROBLEM: RViz not showing map
SOLUTION:
  1. Check Map display is enabled in RViz
  2. Verify /map topic exists: ros2 topic list | grep map
  3. Restart RViz: kill and relaunch
  4. Check RViz config file not corrupted

PROBLEM: Serial port permission denied
SOLUTION:
  sudo usermod -a -G dialout $USER
  newgrp dialout
  (Log out and back in after this change)

PROBLEM: Build fails with module not found
SOLUTION:
  1. Source workspace: source install/setup.bash
  2. Install missing packages: sudo apt install ros-humble-[package-name]
  3. Clean and rebuild: colcon build --packages-select esp32_serial_bridge --symlink-install

================================================================================
EMERGENCY PROCEDURES
================================================================================

TO STOP ROBOT IMMEDIATELY:
  1. Press Ctrl+C in launch terminal (graceful shutdown)
  2. Topic stop: ros2 topic pub /cmd_vel geometry_msgs/Twist --once
  3. Hard stop: Kill all motors (Ctrl+C twice on motor test)
  4. Emergency: Power-cycle ESP32 USB

TO RECOVER FROM FROZEN STATE:
  1. Kill all ROS processes: pkill -9 ros2
  2. Kill RViz: pkill -9 rviz2
  3. Kill SLAM: pkill -9 slam_toolbox
  4. Close serial connection: pkill -f "pio device monitor"
  5. Disconnect USB from ESP32, wait 2 seconds, reconnect
  6. Verify port: ls /dev/ttyACM*
  7. Relaunch system
